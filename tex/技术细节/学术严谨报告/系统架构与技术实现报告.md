## 1. 系统概述

### 1.1 学术定位

本系统的核心学术定位为 **Offline Model-Based Planning on Experience Transition Graphs**（基于经验转移图的离线模型规划）。系统从强化学习（RL）智能体的历史交互轨迹中，统计聚合构建显式的经验转移图（Experience Transition Graph, ETG），并在该图上执行束搜索（Beam Search）进行多步决策规划，同时支持滚动推演（Receding-Horizon Planning）与偏差恢复（Branch-to-Backup Recovery）。

本系统占据了一个独特但完全合理的学术位置——**离线、显式模型、基于搜索的规划**——区别于：

| 范式 | 特征 | 代表方法 | 与本系统的区别 |
|------|------|----------|---------------|
| Online RL | 边交互边学习策略函数 | DQN, PPO | 本系统规划阶段不更新模型 |
| Offline RL | 从离线数据集学习策略函数 | CQL, TD3+BC | 本系统不学习策略函数，直接搜索 |
| Model-Based RL (NN) | 学习神经网络世界模型 | MBPO, Dreamer | 本系统使用表格型显式模型 |
| **本系统** | **离线构建显式转移图 + 图上确定性搜索** | **ETG + Beam Search + BTB** | — |

### 1.2 系统总体架构

系统由两条核心流水线组成：**离线数据管道**（构建 ETG）与**在线实时系统**（ETG 驱动实时决策）。两条流水线通过持久化的 ETG 文件（Pickle 序列化）实现完全解耦。以交互式 Web 可视化平台 (`visualize_kg_web.py`) 为统一入口，集成离线分析、在线决策与实时控制三大功能域。

## 2. 技术流水线

### 2.1 离线数据管道

离线数据管道实现从原始 RL 轨迹到可用 ETG 的完整转换过程，由 `scripts/build_knowledge_graph.py` 驱动：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  原始轨迹数据  │───▶│  数据加载器   │───▶│  状态抽象    │───▶│  ETG 构建    │───▶│  转移网络构建  │
│ (node_log,   │    │ (DataLoader) │    │ (BKTree)    │    │ (DecisionKG) │    │ (transitions)│
│  action_log, │    │              │    │              │    │              │    │              │
│  game_result)│    │              │    │              │    │              │    │              │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
                                              │                    │                    │
                                              ▼                    ▼                    ▼
                                    state_node.txt          KG pickle           transitions pickle
                                    (p,s)→nid 映射          (state_action_map)   (s→a→s'→count)
```

#### 阶段 1：原始轨迹数据

数据来源于 RL 训练过程中智能体与 StarCraft II 环境的交互记录，存储在 `data/{map_id}/{data_id}/` 目录下：

| 文件 | 格式 | 内容 |
|------|------|------|
| `node_log.txt` | 每行一个 episode，状态 ID 空格分隔 | 状态访问序列 |
| `action_log.csv` | 每行一个 episode，动作编码逗号分隔 | 动作执行序列 |
| `game_result.txt` | 每行 `[episode_id, result, score]` | 对局结果（Win=1/Loss=0） |
| `bktree/primary_bktree.json` | JSON 树结构 | 主 BK-Tree（坐标聚类） |
| `bktree/secondary_bktree_*.json` | JSON 树结构 | 次级 BK-Tree（HP 分布聚类） |
| `graph/state_node.txt` | 每行 `(primary_id, sub_id) → nid` | 状态映射表 |

#### 阶段 2：数据加载器（DataLoader）

`src/data/loader.py` 中的 `DataLoader` 类提供懒加载接口，通过 `@property` 装饰器实现按需读取：

- **轨迹数据**：`state_log`（状态序列）、`action_log`（动作序列）、`game_results`（胜负结果）
- **BK-Tree 结构**：`primary_bk_tree`（主树）、`secondary_bk_trees`（按 primary_id 索引的次级树字典）
- **映射表**：`state_node_dict`（状态字符串→nid）、`reverse_dict`（nid→状态元组）
- **距离矩阵**：`state_distance_matrix`（nid 空间的 940×940 距离矩阵）

#### 阶段 3：状态抽象（BK-Tree 层次聚类）

连续状态空间通过两级 BK-Tree 进行离散化：

```
原始观测 obs
     │
     ▼
get_norm_state(obs) → norm_state = {red_army: [...], blue_army: [...]}
     │
     ▼
Primary BKTree (坐标距离) → primary_cluster_id (p)
     │
     ▼
Secondary BKTree[p] (HP 分布距离) → sub_cluster_id (s)
     │
     ▼
state_cluster = (p, s) → state_node.txt → nid (0~939)
```

- **Primary BK-Tree**：基于单位坐标的分布距离（`DistributionDistance`，使用 `scipy.optimize.linear_sum_assignment` 最优匹配），将战场队形聚类为主类别
- **Secondary BK-Tree**：针对每个主类别，基于 HP 分布进一步细分，形成两级层次化状态抽象
- **距离度量**：`DistributionDistance` 类计算两个归一化状态的匹配距离（坐标距离 + HP 差异），单次 940 路比较耗时约 55ms

#### 阶段 4：经验转移图构建（DecisionKnowledgeGraph）

`src/decision/knowledge_graph.py` 中的 `DecisionKnowledgeGraph` 类从轨迹数据构建 ETG：

```
Algorithm: Build ETG from Trajectories
Input: state_episodes, action_episodes, reward_episodes, outcome_episodes
Output: DecisionKnowledgeGraph instance

1. For each episode:
2.   Compute cumulative future rewards (discounted)
3.   For each timestep t:
4.     key = _get_key(state[t], action_history[t-window:t])
5.     kg.add_visit(key, action[t], step_reward[t], future_reward[t], outcome)
6. Return kg
```

每个 `(state, action)` 对维护 `ActionStats` 数据结构：

```python
ActionStats = {
    visits:            int,     # 访问频次 → 近似 P(a|s)
    avg_step_reward:   float,   # 即时奖励均值 → 近似 R(s, a)
    avg_future_reward: float,   # 未来累积奖励均值 → 近似 Q(s, a)
    win_rate:          float,   # 胜率 → 近似策略质量
    quality_score:     float,   # 综合质量评分 = avg_future_reward * 0.7 + win_rate * 30.0
}
```

支持两种查询模式：
- **Simple 模式**：key = state_id，标准 MDP 查询
- **Context 模式**：key = (state_id, history_hash)，高阶 Markov 近似

#### 阶段 5：转移网络构建

`build_transitions()` 函数统计所有状态转移频次，构建转移概率模型：

```
transitions = {
    state_id: {
        action: {
            next_state_id: count    # → P(s'|s,a) = count / sum(counts)
            ...
        }
        ...
    }
    ...
}
```

同时标记终端状态（Win/Loss），用于规划时的终止条件判断。

## 3. 核心数据结构与算法

### 3.1 经验转移图（Experience Transition Graph）

#### 数学定义

经验转移图是一个带权有向图 $G = (V, E, \phi)$：

- **节点集** $V$：抽象状态 ID 集合（$|V| = 940$，对应 MvsM4 场景），由 BK-Tree 两级聚类产生的 $(p, s)$ 对经 `state_node.txt` 映射为整数 nid
- **边集** $E \subseteq V \times A \times V$：动作标签的有向边，$A$ 为离散动作空间（11 个动作，编码为 `a`~`k`）
- **属性函数** $\phi: E \to \mathbb{R}^5$：边上附带的 5 维统计属性

#### 数据结构实现

```
ETG = {
    state_action_map: Dict[int, Dict[str, ActionStats]],   // (state, action) → 统计属性
    transitions:      Dict[int, Dict[str, Dict[int, int]]]  // s → a → s' → count
}
```

#### 统计属性的概率语义

| 属性 | 统计量 | 概率语义 | 近似目标 |
|------|--------|----------|----------|
| `visits` | 访问频次 $N(s, a)$ | $P(a \mid s) \propto N(s, a)$ | MDP 策略 $\pi(a \mid s)$ |
| `transitions` | 转移频次 $N(s, a, s')$ | $P(s' \mid s, a) = N(s, a, s') / N(s, a)$ | MDP 转移函数 $T(s' \mid s, a)$ |
| `avg_step_reward` | 即时奖励均值 $\bar{r}(s, a)$ | $R(s, a) \approx \bar{r}(s, a)$ | MDP 奖励函数 $R(s, a)$ |
| `avg_future_reward` | 未来奖励均值 $\bar{G}(s, a)$ | $Q(s, a) \approx \bar{G}(s, a)$ | MDP 状态-动作值 $Q^*(s, a)$ |
| `quality_score` | 综合评分 $0.7 \bar{G} + 30 \cdot \text{win\_rate}$ | 效用函数 $U(s, a)$ | 综合决策质量 |

### 3.2 BK-Tree 层次聚类

BK-Tree（Burkhard-Keller Tree）是一种满足三角不等式距离度量的层次化数据结构，用于高效将连续状态映射为有限离散状态 ID。

#### 两级聚类架构

```
                    连续状态 norm_state
                          │
            ┌─────────────┴─────────────┐
            ▼                           ▼
    Primary BK-Tree              Secondary BK-Tree[p]
    (坐标分布距离)                 (HP 分布距离)
    距离度量: 坐标匹配              距离度量: HP 匹配
    阈值: τ_primary               阈值: τ_secondary
            │                           │
            ▼                           ▼
    primary_id (p)              sub_id (s)
            │                           │
            └─────────┬─────────────────┘
                      ▼
              state_cluster = (p, s)
                      │
                      ▼
            state_node.txt 查表 → nid ∈ [0, 939]
```

#### 距离度量：DistributionDistance

`DistributionDistance` 类（`src/structure/custom_distance_sc2.py`）计算两个归一化状态的分布距离：

1. 分别提取红蓝两军的坐标矩阵和 HP 向量
2. 计算两军之间的坐标距离矩阵（`np.linalg.norm`）
3. 通过 `scipy.optimize.linear_sum_assignment` 求解最优匹配
4. 返回 `(坐标匹配距离, HP 差异)`

该度量满足 BK-Tree 要求的三角不等式，保证聚类结果的一致性。

#### 运行时只读模式

实时对局中，`KGGuidedAgent` 使用 `query_nearest()` 方法进行只读查询，不向 BK-Tree 插入新节点。这避免了运行时产生训练数据中不存在的 `(p, s)` 组合，确保状态 ID 始终在 ETG 的有效范围内。当查询距离超过阈值时，通过 bridge 发送 warn 级别事件通知前端。

### 3.3 束搜索规划（Beam Search Planning）

#### 算法描述

给定起始状态 $s_0$，在 ETG 上执行受限的多步前向搜索：

```
Algorithm: Beam Search on ETG
Input: s₀, beam_width K, max_depth D, scoring_function f, min_cum_prob θ
Output: BeamSearchResult[] (flat list with parent_idx for tree reconstruction)

1.  Initialize beam B₀ = {(s₀, score=0, cum_prob=1.0, path=[])}
2.  For depth d = 1, 2, ..., D:
3.      Candidates ← {}
4.      For each (s, score, cum_prob, path) ∈ B_{d-1}:
5.          For each action a available at s in ETG:
6.              Get ActionStats φ(s, a) and transitions P(s'|s, a)
7.              If visits(s, a) < min_visits: skip
8.              For each reachable s' with P(s'|s, a) > 0:
9.                  new_cum_prob = cum_prob × P(s'|s, a) × discount^d
10.                 If new_cum_prob < θ: skip
11.                 Add (s', f(s,a)×discount^d, new_cum_prob, path+[(s,a,s')])
12.     Prune: retain top-K by (score, cum_prob) → B_d
13.     If all paths reach terminal states: break
14. Return flat result list (with parent_idx for path reconstruction)
```

#### 评分策略

系统支持三种评分策略：

| 策略 | 评分函数 | 适用场景 | 配置键 |
|------|----------|----------|--------|
| **Quality Score** | $f = \text{quality\_score}(s, a)$ | 综合质量优先 | `quality` |
| **Future Reward** | $f = \text{avg\_future\_reward}(s, a)$ | 长期收益优先 | `future_reward` |
| **Win Rate** | $f = \text{win\_rate}(s, a)$ | 胜率优先 | `win_rate` |

#### 路径评估维度

每条候选路径从四个维度独立评估，综合评分通过 Min-Max 归一化后的加权和计算：

$$\text{CompositeScore} = 0.30 \cdot \hat{C} + 0.30 \cdot \hat{Q} + 0.30 \cdot \hat{W} + 0.10 \cdot \hat{R}$$

其中 $\hat{C}$（置信度）、$\hat{Q}$（质量）、$\hat{W}$（胜率）、$\hat{R}$（奖励）分别为各维度的归一化值。

### 3.4 滚动推演（Receding-Horizon Planning）

#### 单步模式（Single-Step Rollout）

每一步独立执行完整的 Beam Search，从搜索结果中选择最优动作，然后推进一步。算法简洁，但每步都重新搜索，不利用历史搜索信息。

```
For each step t:
    Run Beam Search from current_state
    Select best action a* by action_strategy
    Advance to next_state via transition model
    If terminal: break
```

#### 多步模式（Multi-Step Rollout with BTB Recovery）

多步模式引入 **Plan-Execute-Diverge** 机制，一次规划多步路径，沿路径执行时检测偏差并恢复：

```
Algorithm: Multi-Step Rollout with BTB Recovery
Input: s₀, plan_depth, beam_width, switching_map, score_threshold

1.  current_state ← s₀
2.  While not terminal:
3.      // PLAN: 执行 plan_depth 步束搜索
4.      plan ← Beam Search(current_state, depth=plan_depth)
5.      planned_path ← plan.best_path
6.
7.      // EXECUTE: 沿规划路径逐步执行
8.      For each (s, a) in planned_path:
9.          Execute a, observe actual next_state s'
10.         If s' ≈ planned_next_state (distance < threshold):
11.             Continue along plan
12.         Else:
13.             // DIVERGE: 实际状态偏离规划
14.             If backup_switch enabled:
15.                 Find closest switch point in switching_map
16.                 Switch to backup path via BTB (Branch-to-Backup)
17.                 Re-plan from diverge state
18.             Else:
19.                 Re-plan from actual state immediately
20.             Break execution loop
```

#### BTB 偏差恢复（Branch-to-Backup Recovery）

借鉴 CPU 分支预测中"分支目标缓冲"的设计思想：

1. **预计算**：推演前对 ETG 中每个状态节点预计算备选路径切换点（switching_map）
2. **切换点构建**：评估所有可选动作的路径质量，筛选评分超过阈值的备选路径，按分叉点去重
3. **两阶段匹配**：精确优先（状态 ID 完全匹配）+ 距离回退（连续空间距离阈值 < 0.2）
4. **无缝切换**：命中切换点后直接跳转到备选路径，避免完全重新规划的开销

#### 动作选择策略

| 策略 | 代码键 | 说明 |
|------|--------|------|
| 最优束路径 | `best_beam` | 选择 CumP 最高的束路径的首个动作 |
| 最优子树质量 | `best_subtree_quality` | 选择子树平均质量最高的动作 |
| 最优子树胜率 | `best_subtree_winrate` | 选择子树平均胜率最高的动作 |
| 最高转移概率 | `highest_transition_prob` | 选择转移概率最大的动作 |
| 随机束路径 | `random_beam` | 随机选择一条束路径 |
| ε-贪心 | `epsilon_greedy` | 以概率 ε 随机，否则选最优 |

### 3.6 动作解析与执行（在线推理）

ETG 输出的动作编码需要经过两级解析才能在 SC2 环境中执行：

```
ETG 动作输出: "4d" (两字符编码)
     │
     ▼
_resolve_action("4d")
→ first_char='4' → cluster_method_idx=4 → self._pending_cluster = clusters[4]
→ second_char='d' → action_letter='d' → ACTION_NAME_MAP['d'] → "action_ATK_clu_nearest_weakest"
     │
     ▼
SmartAgent.action_ATK_clu_nearest_weakest(obs)
→ 使用 self.cluster_result (由 _pending_cluster 设置)
→ 执行聚类 + 选择最近最弱目标 + 发起攻击
```

动作编码的设计反映了系统的两级决策结构：第一级（聚类方法）决定如何对敌方单位分组，第二级（作战动作）决定对选定分组执行什么操作。11 个动作覆盖了攻击（ATK）、防御（DEF）、混合（MIX）三大战术类别。

### 5.2 Autopilot 自动驾驶循环

`_autopilot_loop()` 是 Bridge Server 中的异步后台任务，实现 ETG 驱动的实时决策：

```
_autopilot_loop (async, 持续运行)
    │
    ├── 1. 消费事件队列
    │   raw_events = bridge.get_events()
    │   for event: _enrich_event(event) → add_log()
    │   _detect_episode_end(raw_events) → episode tracking
    │
    ├── 2. 获取最新观测
    │   obs = bridge.get_observation(timeout=0.5s)
    │   state_id = _parse_state_id(obs)
    │
    ├── 3. 状态去重
    │   if state_id == _last_decide_state: skip
    │
    ├── 4. 决策计算 (run_in_executor, 避免阻塞事件循环)
    │   action, event_type, plan_snap = _autopilot_decide(state_id)
    │
    ├── 5. 动作发送
    │   bridge.put_action(action)
    │   add_log("success", "autopilot", "执行动作 {action}")
    │   _episode_buffer.append(plan_snap)
    │
    └── 6. await asyncio.sleep(0.01) → 下一轮循环
```

#### 单步模式（single_step）

每帧独立执行完整 Beam Search：

```
_autopilot_decide(state_id) [single_step]
    │
    ▼
_get_plan_from_beam(state_id, lookahead_steps)
→ 返回 (first_action, actions, states, beam_dicts, beam_paths)
    │
    ▼
plan_snap = {
    beam_results: beam_dicts,
    beam_paths: beam_paths,
    mode: "single_step",
    state: state_id,
    ...
}
    │
    ▼
return (first_action, "autopilot_single", plan_snap)
```

#### 多步模式（multi_step）

一次规划多步路径，执行时检测偏差并恢复：

```
_autopilot_decide(state_id) [multi_step]
    │
    ├── 有计划且未执行完 (_plan_idx < len(_action_plan))
    │   ├── next_planned_state = _planned_states[_plan_idx]
    │   ├── _states_match(state_id, next_planned_state, dist_matrix)
    │   │   ├── 匹配: 返回下一个计划动作, _plan_idx++
    │   │   └── 不匹配: 标记 diverge, replan
    │   └── replan: _get_plan_from_beam(state_id, lookahead)
    │
    └── 无计划或已执行完
        └── _get_plan_from_beam(state_id, lookahead) → 新计划
```

### 5.3 对局记录追踪

#### Episode 数据结构

```python
EpisodeRecord = {
    episode_id: int,          # 全局递增 ID
    result: str,              # "Win" / "Loss" / "Unknown"
    score: float,             # 得分
    start_frame: int,         # 起始帧号
    end_frame: int,           # 结束帧号
    plan_count: int,          # 规划次数
    plans: List[plan_snap],   # 每次规划的快照列表
}
```

#### Buffer-Based 提交机制

采用 `end_game_flag` 硬约束 + 缓冲区的方案解决事件队列竞态问题：

```
游戏帧循环:
    _episode_buffer.append({frame, state, action, plan_snap, ...})
    
    if end_game_flag False→True (end_game_flag 过渡):
        _commit_buffer(result="Win"/"Loss")
        → 创建 EpisodeRecord
        → 追加到 _episodes 列表
    
    if score_detected 事件:
        _episodes[-1].score = score  # 仅更新分数，不触发提交
```

**设计理由**：`end_game_flag` 是 observation 级别的布尔信号，直接从游戏观测读取，不经过事件队列，因此不存在竞态条件。

增量日志获取：`after_seq` 参数确保只获取新日志条目，避免重复传输。

### 5.5 游戏循环控制

`run_loop_custom()` 实现支持暂停/恢复/步进/停止的游戏主循环：

```
run_loop_custom():
    For each game step:
        │
        ├── check_control() → 处理控制指令
        │   ├── "pause" → wait_until_resumed()
        │   ├── "step" → 执行一步后自动暂停
        │   └── "stop" → 清理并退出
        │
        ├── check_run_episode() → 单局模式检查
        │
        ├── env.step() → 获取新观测
        │
        ├── 检查 env.f_result (BEFORE last())
        │   ├── "win"/"loss" →
        │   │   ├── kill_all_units()
        │   │   ├── check_run_episode() → 单局模式暂停
        │   │   └── spawn_units() → 新 episode
        │   └── None → 检查 timesteps[0].last()
        │       └── reset_frames 分支 →
        │           ├── kill_all_units()
        │           ├── check_run_episode() → 单局模式暂停
        │           └── spawn_units()
        │
        └── agent.step(obs, env) → KGGuidedAgent 执行决策
```

**关键设计**：`env.f_result` 检查必须在 `timesteps[0].last()` 之前，因为 `f_result` 包含更精确的胜负判定信息。
